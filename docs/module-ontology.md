# WDK Module Ontology

Reference for the internal architecture, module catalog, dependency graph, build system, and extension patterns.

## Architecture Overview

WDK is a zero-dependency browser data engineering toolkit built from standalone JavaScript modules. The build system (`build.js`) concatenates source files in dependency order into a single IIFE bundle. There are no import/export at runtime -- all modules communicate through shared globals within the IIFE closure.

### Deployment Tiers

| Tier | Artifact | Use Case | Size Target |
|------|----------|----------|-------------|
| **Tier 1 -- Bookmarklet** | `dist/wdk-bookmarklet.txt` | Inject into any page via `javascript:` URI | <100 KB |
| **Tier 2 -- Static HTML** | `dist/wdk.html` | Open locally, full app shell with XLSX/SQL/Notebook | ~191 KB |
| **Tier 3 -- Localhost** | `dist/wdk.html` served via HTTP | Full Tier 2 + File System Access API, OPFS, Web Workers | ~191 KB |

All tiers use the same build output. The difference is runtime context: bookmarklet injects into an existing page (uses floating panel), static HTML runs standalone (uses app shell), and localhost gets additional browser APIs from having an HTTP origin.

### Runtime Detection

`main()` (generated by the build) checks for `initWDK()` (app shell). If present, delegates to app shell which auto-detects bookmarklet vs. standalone mode. If absent, falls back to legacy `createPanel()` flow for minimal bookmarklet builds.

### Key Dependency Rules

1. **Parsers are leaf nodes** -- csv.js, json.js, and zip.js depend on nothing. xlsx.js depends only on zip.js.
2. **Transforms depend on parsers** only indirectly (through data passed at runtime).
3. **sql.js depends on sql-functions.js** -- must be loaded after it.
4. **xlsx.js depends on zip.js** -- resolves `unzip` as a global in browser or via `require` in Node.
5. **UI modules depend on transforms/parsers** via globals (e.g., `file-import.js` calls `parseCSV`, `parseJSON`, `parseXLSX`).
6. **app-shell.js depends on all other UI modules** -- must be last in the UI group.
7. **SharePoint modules form a chain** -- sp-auth.js is the foundation; subsequent SP modules depend on its `spFetch`, `spGetDigest` exports.
8. **Inspect modules are self-contained IIFEs** -- they wrap themselves and expose globals; no WDK internal deps.

### Why Inspect is Bookmarklet-Only

Inspect modules (DOM scraper, network interceptor, storage viewer, console capture) operate on the **host page** -- they read the page's DOM, intercept its network calls, and access its storage. In standalone HTML mode, there is no host page to inspect, so these modules serve no purpose.

### Why XLSX/Analysis is Tier 2+

XLSX parsing requires ZIP decompression (~22 KB). SQL and pivot engines add ~46 KB. These are excluded from bookmarklet builds to stay under the 100 KB target for `javascript:` URIs (some browsers enforce URL length limits).

## Build System

### How `build.js` Works

1. **Read sources**: Iterates `SOURCE_FILES` array in order, reading each from `src/`.
2. **Strip module exports**: Removes `module.exports = {...}`, `if (typeof module...)` guards, and `export` keywords so modules work as plain scripts inside the IIFE.
3. **Concatenate**: Joins all source code with separator comments (`// --- parsers/csv.js ---`).
4. **Append `main()`**: Generates the entry-point function that wires modules together.
5. **Wrap in IIFE**: `(function () { "use strict"; ... })();`
6. **Output three artifacts**: `dist/wdk.js` (readable), `dist/wdk-bookmarklet.txt` (URL-encoded `javascript:` URI), `dist/wdk.html` (HTML with JS inlined in `<script>` tag).

### Source Ordering Requirements

The `SOURCE_FILES` array in `build.js` defines concatenation order. This order IS the dependency resolution -- there is no module loader. Rules:

1. `zip.js` must precede `xlsx.js` (xlsx resolves `unzip` as a global)
2. `data-model.js` should precede `pipeline.js` (Pipeline operates on DataFrames)
3. `sql-functions.js` must precede `sql.js` (sql resolves `evalSQLFunction` as a global)
4. All transforms/parsers must precede UI modules (UI calls parser/transform globals)
5. All UI modules must precede `app-shell.js` (shell orchestrates all UI components)
6. `sp-auth.js` must precede other `sp-*.js` modules (they depend on `spFetch`, `spGetDigest`)
7. `sp-compat.js` and `sp-errors.js` must precede `sp-list-*.js` and `sp-doc-*.js`

### Module Stripping Patterns

The build strips these patterns to make CJS/ESM modules work as plain browser scripts:

```javascript
// CJS exports -- removed entirely
module.exports = { foo: foo, bar: bar };

// CJS conditional guard -- removed entirely
if (typeof module !== 'undefined' && module.exports) {
  module.exports = { ... };
}

// ESM export keywords -- keyword stripped, declaration kept
export function foo() {}   -->  function foo() {}
export const bar = 1;      -->  const bar = 1;
export { foo, bar };       -->  (removed)
```

### Skipped Files

If a source file in `SOURCE_FILES` does not exist on disk, the build logs `SKIP (not found)` and continues. This allows the same build config to work across development stages where some modules are not yet implemented.

## Extension Guide

### Adding a New Parser

1. Create `src/parsers/your-format.js`.
2. Expose a parse function as a global (will be available inside the IIFE).
3. Add to `SOURCE_FILES` in `build.js` after existing parsers.
4. Wire into `file-import.js` format detection.

Template:

```javascript
/**
 * WDK YourFormat Parser
 * Zero dependencies.
 */

function parseYourFormat(text, options) {
  // Parse logic here
  // Return DataFrame-compatible: { headers: string[], rows: any[][] }
  return { headers: [], rows: [] };
}

function parseYourFormatFile(file) {
  return new Promise(function (resolve, reject) {
    var reader = new FileReader();
    reader.onload = function () { resolve(parseYourFormat(reader.result)); };
    reader.onerror = function () { reject(reader.error); };
    reader.readAsText(file);
  });
}

if (typeof module !== 'undefined' && module.exports) {
  module.exports = { parseYourFormat: parseYourFormat };
}
```

Then in `src/ui/file-import.js`, add detection in the file handler:

```javascript
} else if (ext === '.yourext') {
  var result = parseYourFormat(text);
  onDataLoaded(new DataFrame(result.headers, result.rows));
}
```

### Adding a SQL Function

Edit `src/transforms/sql-functions.js`. Add to the `SQL_FUNCTIONS` object:

```javascript
your_func: function (args) {
  // args[0], args[1], etc. are resolved argument values
  // Return the scalar result
  return result;
},
```

The function is immediately available in SQL queries as `YOUR_FUNC(...)`. No registration step -- the engine looks up functions by lowercase name in `SQL_FUNCTIONS`.

### Adding a Transform

1. Create `src/transforms/your-transform.js`.
2. Expose functions as globals.
3. Add to `SOURCE_FILES` in `build.js` in the transforms section.
4. Use from UI modules, REPL, or notebook cells.

Transforms should operate on DataFrame-shaped objects (`{ _headers: string[], _rows: any[][] }`) and return the same shape.

### Adding a UI Panel

1. Create `src/ui/your-panel.js`.
2. Export a constructor function: `function createYourPanel(container, getContext) { ... }`
3. Add to `SOURCE_FILES` before `app-shell.js`.
4. Register as a tab in `app-shell.js`:

In app-shell.js, add the tab definition and wire the panel creation:

```javascript
// In the tab definitions array
{ id: 'your-panel', label: 'Your Panel', icon: '...' }

// In the tab content creation logic
case 'your-panel':
  if (typeof createYourPanel === 'function') {
    createYourPanel(contentArea, getContext);
  }
  break;
```

### Adding an Export Format

Edit `src/export/export.js`. Add a conversion function:

```javascript
export function toYourFormat(table, options) {
  options = options || {};
  // table.headers = string[], table.rows = any[][]
  // Return a string representation
  return result;
}
```

Then wire into the UI export controls in `app-shell.js` or wherever export buttons are rendered.

### Adding an Inspector Module

1. Create `src/inspect/your-inspector.js` as a self-contained IIFE.
2. Add to `SOURCE_FILES` at the end of the inspect section.
3. Add to `DK_BUILD_MODULES` in `build-config.js` with `tier: 'inspect'`.

Template:

```javascript
/**
 * INSPECT-NNN: Your Inspector
 * Zero dependencies, var declarations, dk- prefixed CSS.
 */
(function () {
  'use strict';

  function startYourInspector() {
    var log = [];
    // ... inspection logic ...
    return {
      getLog: function () { return log.slice(); },
      clear: function () { log = []; },
      stop: function () { /* cleanup */ }
    };
  }

  // Expose as global
  if (typeof window !== 'undefined') {
    window.startYourInspector = startYourInspector;
  }
})();
```

Conventions for inspect modules:
- Wrap in IIFE (they run in the host page context)
- Use `var` declarations (ES5 compat for government browsers)
- Prefix CSS classes with `dk-` to avoid host page collisions
- Return a control object with `getLog`, `clear`, `stop` methods
- Add to `DK_BUILD_MODULES` in `build-config.js` with `tier: 'inspect'`
